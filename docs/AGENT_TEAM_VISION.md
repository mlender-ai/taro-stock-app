# Agent Team Vision — "하나의 스타트업 팀처럼"

> **이 문서의 위상**: 특정 제품(Trading Taro 등)과 **분리된 독립 비전 문서**다.
> 어떤 제품을 만들든 재사용 가능한 "자율 에이전트 팀" 구축 로드맵이며,
> 언제든 꺼내서 이어서 개발할 수 있도록 작성되었다.
> 제품 코드가 바뀌어도 이 문서의 팀 구조·원칙은 유지된다.

---

## 0. 최종 목표 (North Star of the team)

CEO(창업자)가 **하나의 스타트업 팀과 함께 일하듯** 에이전트들과 협업한다:

- 각 직군 에이전트와 **직접 대화**할 수 있다.
- 에이전트가 슬랙에서 **스스로 안건을 발화**한다.
- 발화된 안건을 두고 관련 직군이 **자동으로 회의**한다.
- 회의 과정과 결정이 **슬랙 채널에 기록**된다.
- 그 기록을 기반으로 에이전트들이 **스스로 학습**한다(회의할수록 똑똑해진다).
- CEO도 그 **의사결정 과정에 참여**한다(끼어들고, 결정하고, 그 결정이 규칙이 된다).

핵심 명제: **에이전트는 "도구"가 아니라 "동료"다.** CEO는 명령자가 아니라 팀의 일원이다.

---

## 1. 왜 3축인가 (비용·노이즈 통제)

9개 직군 에이전트가 매 안건마다 슬랙에서 회의하면 LLM 호출이 폭증하고 채널이 시끄러워진다. 그래서 **처음에는 3축으로 압축**해서 작게 시작한다. "팀처럼 움직인다"는 느낌은 3명으로도 충분히 난다. 이 수준이 안정되면 그때그때 상황에 맞춰 직군을 분화한다.

### 3축 정의 (기존 9 lane 흡수 매핑)

| 축 | 역할 한 줄 | 흡수하는 기존 lane |
|---|---|---|
| **CTO** (개발 전체) | 무엇을 어떻게 만들 것인가 — 구현·기술 결정 | frontend, backend, qa, prompt_engineer |
| **Security** (보안·인프라 전체) | 안전하게·안정적으로 굴러가는가 — 보안·인프라·규제 | security, (인프라/옵저버빌리티), regulation |
| **PM** (제품 중심) | 사용자에게 가치가 있는가 — 제품·우선순위·디자인·시장 | pm, designer, marketer |

> CEO는 4번째 축이 아니라 **팀의 오너**로서 회의에 참여한다(별도 페르소나 `CEO_PERSONA.md` 이미 존재).
> 3축으로 시작하되, 각 축은 내부적으로 흡수한 lane들의 관점을 "겸한다"(예: PM이 디자인·마케팅 관점도 대변). 분화는 나중에.

### 확장 원칙
- **3축이 안정적으로 회의·학습·결정을 돌릴 때까지 직군을 늘리지 않는다.**
- 특정 축의 부하가 크거나 관점 충돌이 잦으면 그때 분화(예: CTO → Frontend/Backend 분리).
- 분화는 "필요가 증명될 때"만. 미리 9개로 벌리지 않는다.

## 2. 전제 — 이 비전은 Phase 0~3 위에 선다

이 팀 비전은 이미 만든/만들 인프라를 재료로 쓴다. **Phase 3(그래프 기억) 완료가 C단계의 선행 조건**이다.

- Phase 0 — 파이프라인 신뢰성(폴백·입력 다이어트). ✅
- Phase 1 — lane-state 원장(완료/킬 이력). ✅ → **회의의 "중복 안건 차단" 규칙으로 재사용**
- Phase 2 — standing constraints(CEO 판정 규칙화). ✅ → **회의의 "금지 영역" + D단계 CEO 결정 규칙화로 재사용**
- Phase 2.5 — 폐루프화(위반 게이트·hits·distill 자동화). ✅ → **회의 발언의 결정론적 검증으로 재사용**
- Phase 3 — 그래프 기억(vault, 관계 그래프). ⏳ → **C단계 "회의가 곧 학습"의 저장소**

즉 새로 0부터 만드는 게 아니라, **기존 조각을 "슬랙 채널 위의 팀 회의"로 재배열**하는 것이다.

## 3. 로드맵 — A·B·C·D

### A. 직군별 독립 정체성 (각자 자기 얼굴로 말하기)
**목표**: 3축이 슬랙에서 *서로 다른 발화자*로 보인다. CEO가 특정 축과 1:1 대화 가능.

- `postMessage`에 `username`(예: "CTO Agent", "PM Agent", "Security Agent")·`icon_emoji`를 넘겨 축마다 다른 정체성으로 발화. (현재는 단일 "Taro Agent Bot" — `lib/slack/client.ts`의 `postMessage`가 정체성 파라미터를 안 보냄.)
- 각 축의 페르소나 파일: `.github/agents/CTO_PERSONA.md`, `PM_PERSONA.md`, `SECURITY_PERSONA.md`(기존 `CEO_PERSONA.md`·`RESEARCHER_PERSONA.md` 형식 따름). 각 페르소나에 "흡수한 lane들의 관점"을 명시.
- 라우팅: CEO가 `@CTO ...` 또는 슬랙 채널/스레드에서 특정 축을 부르면 해당 페르소나로 답. (현재 `handleAgentChat` 단일 페르소나를 축별로 분기.)
- **A만으로도**: "직군과 직접 대화" 니즈 충족.

**병행 가능**: Phase 3와 독립. 먼저 해도 됨.

### B. 슬랙 자율 회의 (발화 → 회의 → 합의 → 기록)
**목표**: 안건이 슬랙 스레드에서 *눈에 보이는 회의*로 처리되고 채널에 남는다.

- 한 축이 안건 발화 → 스레드 생성. 관련 축들이 스레드에 의견. 예: PM "이 기능 필요" → Security "규제 리스크 있음" → CTO "구현 2일".
- **회의 가드레일(필수, 비용·노이즈 통제)**:
  - 라운드 제한: 축당 최대 1발언 + 1반박. 무한 수다 금지.
  - 참여 최소화: 모든 안건에 3축 전부 부르지 않는다. 관련 축만(예: UI 안건엔 PM+CTO, 보안 안건엔 Security+CTO).
  - 중복 차단: Phase 1 lane-state로 "이미 한 회의/킬된 안건" 재회의 금지.
  - 금지 영역: Phase 2 constraints 위반 안건은 발화 단계에서 자동 폐기(Phase 2.5 게이트 재사용).
  - 합의 실패 → CEO 호출(D단계).
- 합의된 결론은 CEO Brief 형식으로 정리(기존 `ceo_brief` 재사용).
- **핵심**: 회의가 채널에 텍스트로 남는다 = "슬랙에 기록" 자동 달성.

**병행 가능**: Phase 3와 독립. A 다음.

### C. 채널 = 학습 입력 (회의가 곧 기억)
**목표**: 회의 기록이 그래프에 적재되고, 다음 회의가 그걸 읽어와 똑똑해진다.

- **Phase 3 그래프 선행 필요.** B에서 채널에 쌓인 회의("이 안건은 이래서 채택/킬")를 vault 노드로 적재.
- 다음 회의 시작 전, 관련 과거 회의 노드를 그래프에서 회수(Graph RAG) → "지난번 비슷한 안건에서 Security가 막았다" 같은 맥락을 자동 주입.
- 회의할수록 그래프 밀도↑ = 회의 품질↑. 측정: 회의당 "과거 노드 인용 수", 반복 안건 감소율.
- **이 단계가 "스스로 학습"의 실체.**

**선행**: Phase 3 완료 후.

### D. CEO 참여형 의사결정 (CEO도 팀의 일원)
**목표**: CEO가 회의에 끼어들고, 그 판단이 일회성이 아니라 팀의 영구 규칙이 된다.

- 합의 실패·고영향 안건 → 봇이 CEO 호출("PM과 Security 의견이 갈립니다. 어느 쪽?").
- CEO가 스레드에 한마디 → 즉시 결정 반영 + standing constraint로 규칙화(Phase 2 메커니즘 재사용).
- CEO가 회의를 *관전만* 할 수도, *주도*할 수도 있게(개입 레벨 선택).
- **결과**: CEO의 판단이 축적되어 팀이 CEO의 멘탈모델에 수렴 = "나와 싱크".

**선행**: B 이후. C와 병행 가능.

## 4. 순서 정리

```
[완료] Phase 0·1·2·2.5
   └─ (현재) Phase 2.5 hits/distill 1~2주 관찰
        └─ Phase 3 (그래프 기억)
             ├─ A. 직군 독립 정체성   ← Phase 3와 병행 가능, 먼저 착수 OK
             ├─ B. 슬랙 자율 회의      ← A 다음
             ├─ C. 회의=학습           ← Phase 3 필요
             └─ D. CEO 참여 결정        ← B 다음, C와 병행
```

- **A·B는 Phase 3 없이도 시작 가능** — "직군과 대화 + 회의가 채널에 기록"까지가 A·B이고, 이것만으로 비전의 약 80%가 채워진다.
- **C·D는 고도화** — 자기학습(C)은 그래프 필요, CEO 참여 규칙화(D)는 B 위에.
- **작게 시작**: A·B를 3축으로 먼저 → 노이즈·비용 감당되는지 확인 → 직군 분화/C·D 확장.

---

## 5. 리스크 & 원칙 (잊지 말 것)

1. **비용·노이즈가 1순위 리스크.** 기능보다 "언제 회의를 *안* 하는가"가 중요. 라운드 제한·관련 축만 소집·중복 차단을 기능 자체보다 먼저 설계.
2. **3축 고정으로 시작.** 안정되기 전 분화 금지.
3. **새 인프라 최소화.** 슬랙 + GitHub + 기존 워크플로로. 별도 DB/큐 도입은 부하가 증명될 때만(기존 North Star 원칙 계승).
4. **모든 회의·결정은 텍스트로 남긴다.** 휘발 금지 — 채널 기록이 곧 C단계 학습 자원.
5. **제품 비종속.** 이 팀 구조는 어떤 제품에도 얹을 수 있게 유지. 제품 특화 규칙은 constraints/페르소나로 주입하되, 팀 메커니즘 자체는 범용.

---

## 6. 착수할 때 (Phase 3 이후)

이 문서를 꺼내 A단계 작업지시서부터 작성한다. 그 시점에 다시 레포를 읽어 현재 `lib/slack/`·`idea-proposal.yml` 구조를 확인하고, A(정체성)→B(회의)→C(학습)→D(참여) 순으로 PR을 쪼갠다. 각 단계는 독립 PR로, 3축으로 작게 검증 후 확장.

> 다음 작업: Phase 2.5 관찰 → Phase 3 → **이 문서 기반 A·B 착수.**
