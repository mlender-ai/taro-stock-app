# Agent Harness Roadmap — "길 잃은 AI를 목표로 되돌리는 장치"

> **이 문서의 위상**: "Pokemon Ralph / Harness Hackathon" 리서치를 우리 시스템에 반추한
> 분석·로드맵 문서다. 구현 청사진이 아니라 **방향 정렬용 기획문서**이며, 특정 베팅을 착수할
> 때 이 문서를 꺼내 thin-slice 스펙을 덧붙인다. 별도 합의 전까지 코드 변경 없음.

---

## 0. 출발 명제 (왜 이 문서)

리서치의 핵심: **"모델이 똑똑하냐"가 아니라 "길 잃은 AI를 목표로 되돌릴 하네스가 있느냐"가
moat다.** 왠만한 모델에게 포켓몬을 맡기면 같은 벽 앞에서 무한 루프를 돈다 — 현재 위치도
다음 목표도 안정적으로 기억하지 못하니까. 차별화는 관찰(observe) → 목표체크(goal-check)
→ 재시도(retry) → 기억(memory) → 복구(recover) 루프를 묶어주는 **하네스**에서 난다.

우리는 방금 자율 에이전트 팀(Stage A·B·D)을 머지했다. 이 렌즈로 우리 시스템을 재평가한다.

**진단 한 줄**: 우리는 하네스 원시기능의 **약 80%를 이미 보유**한다. 그러나 **결과-접지
(outcome) 루프가 없다** — 활동(제안·머지·성공률)은 측정하지만 *진척(North Star 지표 이동)*은
보지 않는다. 포켓몬 비유로: **벽에 안 박게 막는 장치는 있는데, 초록마을 방향인지 확인하는
나침반이 없다.**

## 1. 하네스 5요소 × 우리 시스템 매핑

전수 조사 결과 — 있는 것과 없는 것(갭):

| 요소 | 있음 (보유) | 갭 (없음) |
|---|---|---|
| **Observe (관찰)** | `autonomy-report.yml` 주간 지표(브리핑·Auto PR·성공률) | "막혔는지" 신호 없음 — 활동량만 보고 진척은 안 봄 |
| **Goal-check (목표체크)** | 제안 시점 U/F/N/A 채점 (`.github/agent-scoring-guide.md`, `idea-proposal.yml:1800-1827`) | 결과 시점 체크 없음 — 머지된 PR이 OKR을 실제로 움직였는지 모름 |
| **Retry (재시도)** | 매일 cron 재실행 | 적응형 아님 — 막혀도 "같은 수"를 또 둠(naive agent와 동일) |
| **Memory (기억)** | constraints(금지)·lane-state 원장·feedback 로그·self-history | "금지" 편향 — "뭐가 통했나"(positive playbook) 없음. Stage C(학습) 보류 중 |
| **Recover (복구)** | 제품 런타임 4단 폴백 (`apps/web/lib/tarot/interpret.ts:101-123`) | 회의 "막힘→리셋" 복구 없음 |
| **Loop detection** | 중복 *제안* 차단 (`ceo-brief-fallback.ts:91-134`, `build-lane-state.ts`, `constraints.ts:316`) | 메타 루프 미탐지 — "시스템 전체가 헛돌고 있다"는 못 잡음 |

요약: 우리는 "한 수 한 수가 규칙 위반인지"는 깐깐하게 본다. 그러나 *"이 게임을 이기고 있는가"*는
아무도 안 본다.

## 2. 베팅 (Impact × Effort)

```
 High │  B4 제품 AI         B1 결과-접지 루프 ★
Impact│  자가검증·교정       (키스톤)
      │                     B2 막힘 감지 + 재접지
      │  ───────────────────────────────────────
 Low  │  (관측 대시보드)     B3 Positive Playbook
      │                     (Stage C-lite)
      └───────────────────────────────────────────
         Low Effort                 High Effort
```

- **B1 — 결과-접지 목표 루프 (키스톤)**: 머지된 Auto PR의 "측정지표"(이미 제안에 포함)를
  머지 시점에 스냅샷 → 7일 후 재측정 → "이 베팅이 지표를 움직였나" 판정을 memory에 적재.
  *이게 없으면 B2·B3가 신호원이 없어 작동 못 한다.*
- **B2 — 막힘 감지 + 자동 재접지**: "N사이클·X PR 머지했는데 핵심 OKR 지표 평평 +
  반복률↑ → STUCK" 신호 → "다른 각도로 제안하라/목표를 의심하라" 프롬프트 주입 또는 CEO
  STUCK 리포트 에스컬레이션. = 포켓몬 "벽 루프 → 복구".
- **B4 — 제품 AI 자가검증·교정 (런타임)**: 타로 해석이 *실제 지표와 일치하는지* 접지 검증
  (RSI 60인데 "과매도" 환각 차단) + 폴백 전 1회 자가교정 재시도. 현재는 금칙어(catastrophic)만
  잡고 "그럴듯하지만 틀린" 출력은 그대로 출고. **사용자 신뢰·규제·리텐션 직결.**
- **B3 — Positive Playbook (Stage C-lite)**: "성공한 결과"를 재사용 패턴으로 distill →
  다음 제안에 주입. 보류한 Stage C(회의=학습)의 가벼운 선행 버전.

## 3. 로드맵 (두 영역 모두)

### 영역 1 — 어드민/운영 에이전트 (회의 팀 하네스)

```
B1 결과-접지 루프 (키스톤)
   └─ B2 막힘 감지 + 재접지      (B1 신호로 STUCK 판정 → 재접지 / CEO 에스컬레이션)
        └─ B3 Positive Playbook  (성공 패턴 distill → 다음 제안 주입 = 보류된 Stage C 선행)
```
**재사용 자산**: `autonomy-report.yml`, 제안 측정지표 필드, `ceo-brief-fallback.ts` 반복탐지,
`constraints.ts` / `build-lane-state.ts` 원장, feedback 로그.

### 영역 2 — 제품 런타임 AI (타로 해석 하네스)

```
B4 자가검증·교정
   └─ (확장) 출력 품질 관측      (source/품질 점수를 TarotDrawHistory에 적재 → 대시보드)
```
**재사용 자산**: `apps/web/lib/tarot/interpret.ts` 4단 폴백 체인,
`checkSafety` (`packages/tarot-core/src/safety/forbidden.ts`), `TarotDrawHistory` 스키마.

### 착수 우선순위 (제안)

1. **B4** — 사용자 신뢰·규제·리텐션 직결, 얇게 시작 가능(접지 검증 1차).
2. **B1** — 운영 자율성의 키스톤. B2·B3의 신호원이라 먼저 깔아야 확장 가능.
3. **B2 → B3** — B1 이후 운영 영역 고도화.

## 4. 착수할 때

특정 베팅을 개발하기로 하면, 그 시점에 레포를 다시 읽어 위 재사용 자산의 현재 구조를 확인하고
**해당 베팅의 thin-slice 구현 스펙(파일·재사용·검증 절차)을 이 문서 하단에 추가**한 뒤 착수한다.
각 베팅은 독립 PR로, 얇게 검증 후 확장한다.

> 다음 후보: **B4(제품 AI 자가검증)** 또는 **B1(운영 결과-접지 루프)**.
