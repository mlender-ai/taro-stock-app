# 데이터 엔진 고도화 전략 (DATA_ENGINE_STRATEGY)

| | |
|---|---|
| **문제** | 제품이 보여주는 정보의 품질이 표면적이다. "그래서 뭐?"가 안 풀린다. |
| **합의** | UI·UX보다 백엔드 데이터 엔진 고도화가 선행해야 한다. |
| **선행 문서** | PRODUCT_TRUTH / KEYWORD_ENGINE_SPEC / PRODUCT_VISION |
| **이 문서** | 엔진을 "수집-분류-카운팅"에서 "수집-이해-재가공-응축"으로 끌어올리는 단계 플랜 |

---

## 1. 진단 (코드 기반, 현재 상태)

제품의 가치 = **파편화된 정보를 응축해 스스로 판단하게 한다**(PRODUCT_VISION §1).
지금 엔진은 그 약속을 못 지킨다. 코드를 뜯어본 결과:

| 단계 | 가치 약속 | 현재 구현 | 갭 |
|---|---|---|---|
| **수집** | 토스·트위터·텔레그램·뉴스·수급 등 파편을 한 곳에 | Yahoo 뉴스 1 + 네이버 종목 4 + 레딧 1 | 사용자가 실제 쓰는 소스 대부분 미연동 |
| **이해** | 기사·여론을 읽고 의미 파악 | 사전 키워드 매칭("반도체" 단어 있나 검사) | 내용 이해 0. 빈도 카운팅 수준 |
| **재가공** | "사람들이 뭐라는지" 응축 | 커뮤니티 글 제목 버리고 개수만 셈 | 여론의 내용이 사라짐 |
| **응축** | "다 확인했다"는 안도감 | 뉴스 제목 3개 나열 | "그래서 뭐?"가 안 풀림 |

**핵심 결론**: UI는 충분하다. 보여줄 **알맹이(가공된 인사이트)**가 없는 게 병목이다.
점수·코멘트·카드는 다 "표면"이고, 그 아래 "이해·재가공" 레이어가 비어 있다.

---

## 2. 전략 프레임 (BCG식 — 어디에 투자할 것인가)

가치사슬을 4단계로 보면, 투자 우선순위는 **뒤쪽(이해·재가공)이 아니라 가운데**다.

```
수집 ────→ 이해 ────→ 재가공 ────→ 응축(표현)
 (넓힌다)   (핵심 해자)   (핵심 해자)    (이미 됨)
   ②          ①            ①            ✓
```

- **이해·재가공(①)이 진짜 해자다.** 수집은 누구나 한다(토스도 뉴스 모은다). 차별점은 "모은 걸 어떻게 *이해하고 재가공*하느냐"다. 여기가 LLM 에이전트가 빛나는 곳이고, 경쟁사(스톡베리=자문, 인사이트업=교육, koreant=밈)가 안 하는 칸이다.
- **수집(②)은 넓히되 과투자 금지.** 소스를 늘리는 건 필요하지만, 소스만 늘리고 이해가 얕으면 "더 많은 표면 정보"가 될 뿐이다. 수집은 이해 레이어가 소화할 수 있는 만큼만.
- **응축(✓)은 이미 충분.** UI·카드·코멘트는 손대지 않는다(PRODUCT_VISION 고정 제약).

> **한 줄 전략: 소스를 적당히 넓히고, 그 위에 "이해·재가공 에이전트 레이어"를 얹어 해자를 만든다.**

---

## 3. 아키텍처 비전 — "자율 인사이트 파이프라인"

사용자가 첨부한 레퍼런스(Firecrawl+Slack 자동 리포트, 옵시디언 세컨드브레인, 자율 에이전트)가 가리키는 방향:
**뒷단에서 에이전트가 스스로 긁고·이해하고·재가공해 "판단 데이터"를 쌓는 구조.**

```
[1. 수집 레이어]  자율 스크래퍼 (cron + Firecrawl류)
   뉴스 RSS · 네이버 종토방 · 트위터/X · 텔레그램 공개채널 · 수급 데이터
        ↓ (원문 보존 — 제목 버리지 않음)
[2. 이해 레이어]  LLM 에이전트가 원문을 *읽고* 구조화
   - 이 기사/글이 무슨 종목·섹터·이슈인가 (분류 아닌 이해)
   - 핵심 주장은? 강세/약세 근거는? 사람들의 실제 워딩은?
   - 출처·신뢰도·발행시점
        ↓ (구조화된 "판단 재료" JSON)
[3. 재가공 레이어]  에이전트가 응축
   - 같은 종목/테마의 여러 소스를 교차 → "오늘 이 종목 왜 떴나" 한 문단
   - 강세론 vs 약세론 균형 정리 (한쪽만 X)
   - "사람들이 체크한 것들" = 섹터·차트·여론·수급을 한 카드에
        ↓ (지식 베이스에 누적 — 옵시디언식 vault)
[4. 응축/표현 레이어]  기존 카드 UI (변경 없음)
   - 점수(보조지표) + 응축된 판단 재료 + 동질감 한 줄
```

핵심 전환: **2·3 레이어가 새로 생기는 것.** 지금은 1(빈약) → 4(직행)이라 표면적이다.
2·3을 에이전트로 채우면 "그래서 뭐?"가 "아 이래서 떴고, 이게 강세 근거, 이게 약세 근거구나"로 바뀐다 = 안도감("다 확인했다").

## 4. 단계 플랜 (실리콘밸리식 — 작게 증명하고 확장)

큰 빅뱅 금지. "하나의 종목/테마로 깊게 증명 → 폭 확장"이 정석.

### Phase A — 이해 레이어 PoC (가장 먼저, 가장 중요)
**목표**: 소스 1~2개라도 "카운팅"이 아니라 "이해"로 바꾼다.
- 기사 원문을 LLM 에이전트가 읽고 구조화: {종목, 섹터, 강세근거, 약세근거, 핵심워딩, 출처, 시점}.
- 커뮤니티 글도 제목·본문 보존해서 같은 구조로.
- **증명 기준(User Zero)**: 광혁이 직접 보고 "이 정도면 내가 따로 안 찾아봐도 되겠다" 싶은가.
- 범위: 딱 한 테마(예: 반도체)로 깊게. 폭은 나중.

### Phase B — 재가공/응축 레이어
**목표**: 구조화된 재료를 "한 카드 분량"으로 응축하는 에이전트.
- 같은 테마의 여러 소스 교차 → "왜 떴나" 한 문단 + 강세/약세 균형 + 사람들 워딩 요약.
- 균형추(양쪽 다)·투자조언 금지 가드는 그대로.
- **증명 기준**: 뎁스 페이지가 "뉴스 제목 3개"에서 "응축된 판단 재료"로 바뀌는가.

### Phase C — 수집 레이어 확장
**목표**: 이해 레이어가 소화 가능한 만큼 소스를 넓힌다.
- 우선순위: 토스 종토방 여론 > 트위터/X > 텔레그램 공개채널 > 수급 데이터.
- 자율 스크래퍼(cron). Firecrawl류로 본문 추출, 실패 시 재시도·폴백(정직 원칙).
- **주의**: 소스만 늘리고 이해가 안 따라가면 "표면 정보 더미"가 된다. A·B 먼저.

### Phase D — 지식 누적 / 자율 학습
**목표**: 매일 쌓인 판단 재료를 vault(옵시디언식)에 누적 → 시계열·맥락.
- "어제 이 종목 이랬는데 오늘 이렇게 됐어"(연속성) 가능해짐.
- 에이전트가 과거 데이터 참조해 "지난번 비슷한 상황" 응축.
- 흔들림 점수·개인화(PRODUCT_VISION Phase 5~6)의 진짜 연료가 여기서 나옴.

## 5. 가내수공업 단계 (지금 당장의 현실 전략)

초기엔 유저가 적으니 **사람(User Zero)이 에이전트를 보조**한다.
- A·B 레이어가 뽑은 응축본을 광혁이 직접 검수·교정 → 품질 기준선 확보.
- 그 교정 데이터가 곧 에이전트 프롬프트·룰 개선의 입력(휴먼 인 더 루프).
- 구독자 한정으로 손수 품질 챙기는 것도 OK(스케일보다 품질 증명이 먼저).
- 목표: "에이전트 단독으로도 광혁 검수 수준"에 도달하면 자동화 비중↑.

## 6. 안 할 것 / 지킬 것

- **UI·카드·스와이프 변경 금지**(PRODUCT_VISION 고정 제약). 이 문서는 *뒷단*만.
- **투자조언·매매신호 금지** 그대로. 이해·재가공은 "판단 재료"지 "답"이 아니다.
- **정직한 숫자·정직한 빈 상태** 그대로. 가공 실패 시 위장 금지, confidence 노출.
- **균형추 필수**: 재가공이 강세론만 응축하면 안 됨. 강세/약세 둘 다.
- **소스 과투자 금지**: 이해 레이어(A·B) 없이 수집(C)부터 늘리지 않는다.
